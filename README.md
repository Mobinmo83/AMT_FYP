# Piano AMT — Onsets & Frames + decoder post-processing 

A PyTorch piano transcription pipeline built on the *Onsets and Frames* model (Hawthorne et al., 2018, 2019) and trained on MAESTRO v3.0.0 (Hawthorne et al., 2019), with a controlled study of decoder-stage post-processing methods. Solo piano audio in → MIDI out: any input audio is resampled to 16 kHz mono, converted to a 229-bin log-mel spectrogram at 31.25 fps, fed through a four-head CNN + BiLSTM that predicts onset, frame, offset, and velocity rolls over the 88 piano keys, and decoded into note events. The repository accompanies a final-year undergraduate dissertation; the report itself contains the methodology, related work, and full results discussion. This README serves as both the project overview and the runbook.

## Architecture summary

`OnsetsAndFrames` (default `model_complexity = 48` → internal `model_size = 768`, **26,491,256 trainable parameters**) takes a `(B, 229, T)` log-mel spectrogram and returns five output tensors of shape `(B, T, 88)`. The four detection heads (Onset, Offset, Frame acoustic, Frame combined) are trained with binary cross-entropy; the Velocity output uses a masked mean-squared-error objective and is predicted only at onset positions. The architecture follows the MAESTRO-paper variant of Onsets and Frames (Hawthorne et al., 2019), which extends the original onset/frame formulation of Hawthorne et al. (2018) with an explicit offset head and an increased model capacity, and is implemented using the open-source PyTorch reference of Kim (2019) as a software template.

| Head | Architecture | Output |
| --- | --- | --- |
| Onset | ConvStack → BiLSTM → Linear → Sigmoid | onset prob ∈ [0, 1] |
| Offset | ConvStack → BiLSTM → Linear → Sigmoid (separate weights) | offset prob ∈ [0, 1] |
| Frame (acoustic) | ConvStack → Linear → Sigmoid (no BiLSTM) | acoustic frame prob ∈ [0, 1] |
| Frame (combined) | BiLSTM → Linear → Sigmoid over `cat(onset.detach(), offset.detach(), frame_acoustic)` | final frame prob ∈ [0, 1] |
| Velocity | ConvStack → Linear (no Sigmoid) | raw, masked-MSE target |

Onset and offset are detached before they enter the combined frame stack so the frame loss does not leak gradients back into the onset and offset detectors. Audio preprocessing constants (`SAMPLE_RATE = 16000`, `HOP_LENGTH = 512`, `N_MELS = 229`, `FRAMES_PER_SECOND = 31.25`, `N_KEYS = 88`, `MIN_MIDI = 21`, `MAX_MIDI = 108`) are fixed across the pipeline and asserted by `piano_amt/scripts/verify_pipeline.py`.

## Decoder post-processing methods

Seven post-processing methods are implemented in `piano_amt/models/onsets_frames/decode_advanced.py` on top of the baseline decoder `piano_amt/models/onsets_frames/decode.py`. Each is gated by an explicit keyword argument so any combination can be evaluated. Methods were experimented with individually and in combination on the MAESTRO validation split (Hawthorne et al., 2019); three preset combinations are exposed in the demo notebook, and the post-processing notebook lets you toggle and tune any method solo or together. The motivation, lineage and ablation results for each method are discussed in the report; brief lineage notes are given in the table below.

| # | Toggle keyword | What it does |
| --- | --- | --- |
| M1 | `use_onset_conditioned_offset` | Lets the trained offset head terminate notes alongside the frame head, related to offset-aware high-resolution decoding (Kong et al., 2021). |
| M2 | `use_frame_smoothing` | Median / Gaussian / morphological-closing smoothing of the frame roll before binarising. |
| M3 | `min_note_duration_ms` | Drops decoded notes shorter than the configured millisecond threshold (default 16 ms). |
| M4 | `use_duplicate_removal` | Within each pitch, merges onsets that fall within `duplicate_tolerance_sec` and keeps the loudest. |
| M5 | `use_chord_grouping` | Snaps onsets within `chord_tolerance_sec` to the median (or earliest) onset of the cluster. |
| M6 | `use_adaptive_thresholds` | Per-piece thresholds set as `base + k·std(activations)`. |
| M7 | `use_pedal_extension` | Extends offsets that fall on frames where many pitches are simultaneously active (sustain-pedal heuristic, related to but simpler than the trained pedal head of Kong et al., 2021). |

Application order inside `advanced_rolls_to_note_events`: M6 → M2 → (M1 + M3) → M4 → M5 → M7. Named presets in [`piano_amt/demo/decoder_presets.py`](piano_amt/demo/decoder_presets.py) ship the report's chosen combinations: `baseline`, `efficient_m3_m4` (M3 + M4), and `quality_m2_m3_m4` (M2 + M3 + M4).

## 6. Hardware notes

- **Demo notebook.** A free Colab T4 GPU is sufficient. CPU works but is much slower.
- **Training subsets ≤ 800 files.** A T4 (16 GB) is fine.
- **Full-piece evaluation** (whole-piece BiLSTM forward pass) needs more headroom because the longest MAESTRO pieces exceed 30,000 mel frames. The released checkpoint was evaluated on an `NVIDIA RTX PRO 6000 Blackwell Server Edition` (102 GB). Efficient decoding runs at roughly 3–4 s per piece; the quality `m2_m3_m4` profile is slower because frame smoothing adds post-processing cost and runs at roughly 8–9 s per piece. cuDNN's LSTM kernel additionally fails on very long sequences (`CUDNN_STATUS_NOT_SUPPORTED`), so `evaluate.py` and `evaluate_advanced.py` disable cuDNN inside the eval forward pass via `torch.backends.cudnn.flags(enabled=False)` and use the native PyTorch LSTM kernel.
- **AMP + TF32.** `Trainer` enables `torch.amp.autocast("cuda")` and TF32 matmul automatically when `cuda.get_device_capability(0)[0] >= 8` (Ampere or newer).
- **`torch.compile`.** Enabled by default in `Trainer` on PyTorch 2.0+ (CUDA only). Disable by passing `use_compile=False` if it triggers backend issues. State-dict save/load handles the `_orig_mod` attribute automatically.
